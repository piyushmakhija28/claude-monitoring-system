# ADR-003: GPU-First Multi-Provider LLM Routing with Fallback Chain

**Status:** Accepted
**Date:** 2026-03-20
**Deciders:** Pipeline Architecture Team

---

## Context

The pipeline's first version was hard-wired to Anthropic's Claude API. This
created three operational problems:

1. **Cost concentration:** All 10-15 LLM calls per run hit Anthropic at
   pay-per-token rates. For high-frequency development sessions this became
   expensive (~$0.30-0.80 per full pipeline run with Claude Sonnet).

2. **Availability risk:** When Anthropic's API returned 529 (Overloaded) during
   peak hours, the entire pipeline stalled with no fallback path.

3. **Latency for simple steps:** Steps like Step 14 (summary) or Step 8 (issue
   label classification) are low-complexity and do not need a frontier model.
   Routing them to a 7B local model saves ~$0.02 per call and ~1-2 seconds of
   network round-trip.

The team needed a routing layer that: (a) uses local GPU inference when available
to reduce cost and latency, (b) falls back to remote providers gracefully, and
(c) selects the right model tier for each step's complexity.

---

## Decision

Implement a GPU-first, cost-aware fallback chain across four providers in
`llm_mcp_server.py`:

```
1. Anthropic (Claude Sonnet / Haiku)   -- remote; highest quality
2. OpenAI (GPT-4o / GPT-4)            -- remote; backup frontier
3. Ollama (qwen2.5:7b)                 -- local GPU; zero marginal cost
4. NPU (on-device inference)           -- local; lowest latency; smallest model
```

### Routing logic

Each step is tagged with a complexity tier:

| Tier | Steps | Primary Provider |
|------|-------|-----------------|
| High (8-10/10) | step2 (plan), step7 (final prompt), step11 (review) | Anthropic Opus/Sonnet |
| Medium (5-7/10) | step0, step3, step5, step10 | Anthropic Haiku or OpenAI GPT-4o |
| Low (1-4/10) | step1, step8, step13, step14 | Ollama qwen2.5:7b (local) |

Complexity is dynamically adjusted by the call graph: if `analyze_impact_before_change`
returns `risk_level=high`, the step is promoted to the tier above.

When a provider call fails (timeout, 4xx/5xx, quota exhausted), the router
automatically tries the next provider in the chain without surfacing the error
to the calling node. The node only sees a failed call if all four providers
exhaust.

The active provider can be forced for debugging: `FORCE_LLM_PROVIDER=ollama`.

### Async health checking

`llm_mcp_server.py` runs a background health check every 60 seconds that pings
all four providers and updates an in-memory availability map. When a provider
recovers, it re-enters the rotation immediately.

---

## Alternatives Considered

| Option | Pros | Cons |
|--------|------|------|
| **Single provider (Anthropic only)** | Simple; predictable | High cost; single point of failure |
| **LiteLLM proxy** | Unified API; many providers | External service to operate; adds network hop; version coupling |
| **vLLM server** | High throughput; batching | Heavy infra; requires A100-class GPU; overkill for single-user dev |
| **OpenRouter** | Managed routing; cost transparency | Paid intermediary; data passes through third party; latency added |
| **Custom router (chosen)** | Full control; no extra service | Implementation burden; must maintain provider SDKs |

LiteLLM was the strongest alternative. It was rejected because it requires
running a proxy server, which adds an ops dependency and a network hop.
The custom router in `llm_mcp_server.py` is ~300 lines and gives full control
over the fallback order, health checking, and per-step tier assignment.

---

## Consequences

**Positive:**
- ~60% reduction in Anthropic API spend for typical development sessions (low-tier
  steps now run on local Ollama).
- Pipeline is resilient to individual provider outages — only loses capability
  if all four providers fail simultaneously.
- NPU path (when available) provides sub-second inference for Step 14 summaries
  and Step 8 label classification.
- `FORCE_LLM_PROVIDER` makes provider-specific debugging trivial.

**Negative:**
- Ollama must be running and the `qwen2.5:7b` model pulled for the local fallback
  to work. A fresh developer machine has neither.
- Quality variance: a Step 2 plan generated by Ollama (on Anthropic outage) is
  lower quality than one from Claude Sonnet. The pipeline cannot currently detect
  this quality drop.
- The 60-second health check interval means a provider that recovers after a
  short outage may not re-enter rotation for up to 60 seconds.
- NPU support requires hardware availability and driver installation — most CI
  environments will not have NPU access.

**Risks:**
- If `FORCE_LLM_PROVIDER` is left set in production after a debugging session,
  the pipeline will use a suboptimal provider for all steps.
  Mitigation: document the env var in the deployment guide; add a startup warning
  log when it is set.
