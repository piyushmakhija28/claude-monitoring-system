# v1.14.0 Design Brief — Orchestrator Input for solution-architect

## CONTEXT DUMP (read before producing the design document)

### What the pipeline looks like now (v1.13.0)

```
Pre-0 (orchestration_pre_analysis_node)
  -- call graph hot/leaf scan -> call_graph_metrics in state
  -- RAG orchestration lookup (0.85 threshold) -> rag_orchestration_hit
  -- template fast-path if --orchestration-template flag given
    if fast-path or RAG hit -> route to step8 (skip Step 0)
    else -> step0_0 (project context pre-flight)

Step 0.0 (step0_0_project_context_node)  -- reads README, CHANGELOG, pyproject.toml, etc into step0_0_project_context

Step 0.1 (step0_1_initial_callgraph_node)  -- snapshot_call_graph() -> step0_1_initial_callgraph for Step 11 diff

Step 0 (step0_task_analysis_node)
  -- Reads combined_complexity_score (1-25) from state (set by Level 1)
  -- Reads call_graph_metrics.complexity_boost from pre-0 output
  -- Calls analyze_impact_before_change(project_root, target_files, task_desc)
  -- Builds _augmented_state with complexity + call graph injected
  -- Calls _run_step(0, ..., step0_task_analysis, _augmented_state)
    --> step0_task_analysis() calls call_execution_script("prompt-generator", args, model_tier="fast")
    --> call_execution_script runs subprocess: python architecture/prompt-generator.py
    --> That subprocess is the single LLM call #1 in v1.13.0
  -- Post-call: populates step1_*, step3_*, step4_*, step5_*, step6_*, step7_* fields
  -- Writes session_dir/system_prompt.txt

Step 2 (step2_plan_execution_node) -- retained, conditional
Steps 8-14 (GitHub issue, branch, impl, PR, close, docs, summary) -- unchanged
```

### LLM call mechanism (how calls happen now)

- `call_execution_script(script_name, args, model_tier)` in `level3_execution/helpers.py`
- Runs `subprocess.run([sys.executable, str(script_path)], ...)`
- The subprocess is a standalone Python script in `level3_execution/architecture/` that makes its own LLM call (likely via mcp-llm-provider or mcp-anthropic-provider)
- Returns JSON parsed from stdout

### What v1.14.0 wants to replace Step 0 with

```
NEW Step 0:
  build_prompt_gen_input()
    -- assembles: system_prompt_template (constant/file)
                  + user's plain text requirements (state.user_message)
                  + RUNTIME CONTEXT block (call_graph_metrics + combined_complexity_score)
  call_prompt_generation_expert(assembled_input)
    -- LLM Call #1 -> returns filled orchestration prompt (string)
    -- stores in state as: orchestration_prompt (new field)
  call_orchestrator_agent(orchestration_prompt)
    -- subprocess call -> executes full plan
    -- stores result in state as: orchestrator_result (new field)
    -- this subprocess streams output to conversation (verbose progress)
Steps 8-14 continue unchanged
```

### Key constraint: verbose progress output

The user must see step-by-step progress in the Claude conversation stream in real time.
The existing mechanism for this is print() to sys.stderr (which the pipeline already does).
The hooks infrastructure:
  - post_tool_tracker runs on every PostToolUse event -> tracks progress
  - stop_notifier runs on every Stop event -> session maintenance + voice

The question is: how does a subprocess call produce output that appears in the Claude conversation stream?
