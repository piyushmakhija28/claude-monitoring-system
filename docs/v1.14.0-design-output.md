## A. Current State Summary

### What Pre-0 Produces and Stores in State

`orchestration_pre_analysis_node` (in `nodes/orchestration.py`) runs before any Step 0 node and writes:

| State Field | Type | Description |
|---|---|---|
| `call_graph_metrics` | Dict | hot_nodes list, leaf_nodes list, dependency_order, complexity_boost int (+1 to +3) |
| `pre_analysis_result` | Dict | Raw output from `get_orchestration_context()` (same as call_graph_metrics) |
| `rag_orchestration_hit` | bool | True if RAG confidence >= 0.85 |
| `rag_orchestration_confidence` | float | RAG match score |
| `rag_orchestration_cached_plan` | Dict | Full cached step0-6 plan if RAG hit |
| `skip_architecture` | bool | True on RAG hit or template fast-path |
| `skip_consensus` | bool | True on RAG hit or template fast-path |
| `template_fast_path` | bool | True if --orchestration-template flag was provided |
| `pre_analysis_execution_time_ms` | float | Wall-clock ms |

On template fast-path or RAG hit, Pre-0 also writes all step0/1/3/5/6/7 migration fields directly, then routes to Step 8, bypassing Step 0 entirely.

### What Current Step 0 Does (v1.13.0)

Step 0 is implemented in two layers:

**Wrapper layer (`nodes/step_wrappers_0to4.py` — `step0_task_analysis_node`)**
1. Reads `combined_complexity_score` from state (1-25 scale, set by Level 1)
2. Calls `analyze_impact_before_change(project_root, target_files, task_desc)` for call graph risk
3. Merges complexity + call graph data into `_augmented_state`
4. Calls `_run_step(0, "Task Analysis + Orchestration Template", step0_task_analysis, _augmented_state, ...)`
5. Post-call: populates step1/3/4/5/6/7 migration fields using setdefault (template output takes precedence)
6. Writes `session_dir/system_prompt.txt`
7. Returns merged result dict

**Step function layer (`steps/step0_task_analysis.py`)**
1. Reads `user_message` from state
2. Builds `context_data` dict (files loaded, session info, patterns, project)
3. Calls doc detection (Level3DocumentationManager.detect_project_docs)
4. Calls `call_execution_script("prompt-generator", args, model_tier="fast")` — **LLM Call #1**
5. Gets `task_type`, `complexity`, `reasoning` from result
6. Calls `call_execution_script("task-auto-analyzer", args, model_tier="fast")` — **LLM Call #2**

Wait — v1.13.0 actually makes TWO subprocess calls (prompt-generator + task-auto-analyzer), not one. The CLAUDE.md says "~1 LLM call" which refers to the consolidated template approach, but the actual step0_task_analysis.py still calls two scripts. This means the "single LLM call" claim in CLAUDE.md describes the intent/architecture of the orchestration template, not what the code currently does.

**Fields written by current Step 0:**
- `step0_task_type`, `step0_complexity`, `step0_reasoning`, `step0_tasks`, `step0_task_count`
- `step0_docs_found`, `step0_target_files` (from doc detection)
- `step0_call_graph_risk_level`, `step0_call_graph_danger_zones_count`, `step0_call_graph_affected_methods_count`
- `step0_complexity_injected`
- `step7_execution_prompt`, `step7_prompt_saved`, `step7_system_prompt_file`
- (via setdefault) `step1_plan_required`, `step3_tasks_validated`, `step4_model`, `step5_skill/agent/skills/agents/skill_definition/agent_definition`, `step6_skill_ready/agent_ready/validation_status`

### How the LLM Gets Called Currently

- Mechanism: `call_execution_script(script_name, args, model_tier)` in `level3_execution/helpers.py`
- Implementation: `subprocess.run([sys.executable, str(script_path)], capture_output=True, text=True, timeout=30, ...)`
- The subprocess is a standalone `.py` script in `level3_execution/architecture/` directory
- Scripts make their own LLM calls (via mcp-llm-provider or mcp-anthropic-provider MCP server tools)
- Returns: JSON parsed from stdout
- Model tier: passed via `MODEL_TIER` env var to the subprocess

**Key architectural fact:** Every LLM call in the pipeline goes through a subprocess + script pattern. There is no direct LLM client instantiation in the pipeline nodes themselves. This is the established pattern v1.14.0 must follow.

### System Prompt Template Storage Decision

**Recommendation: Option B — text file at `level3_execution/templates/orchestration_system_prompt.txt`**

Rationale:
- Python constants (Option A) make the template hard to edit without touching Python code — poor for iteration
- `.md` file (Option C) looks appealing but introduces markdown rendering ambiguity when injected as a prompt
- `.txt` file (Option B) is unambiguous, human-readable, and matches the existing pattern (`session_dir/system_prompt.txt`)
- Can be read with `Path(...).read_text(encoding="utf-8")` at call time — no import needed
- Allows future template versioning by adding `orchestration_system_prompt_v2.txt` without code changes

**Template structure (logical sections):**
```
## ROLE
You are an orchestration expert...

## MY REQUIREMENTS
{user_requirements}

## RUNTIME CONTEXT
{runtime_context_json_block}

## OUTPUT FORMAT
Return a fully-formed orchestration plan in this exact JSON structure:
{...}
```

The `{user_requirements}` and `{runtime_context_json_block}` are the only two injection points. Everything else is static.

### Where RUNTIME CONTEXT Gets Appended

RUNTIME CONTEXT is not appended — it is injected inline into the template at a fixed `{runtime_context_json_block}` placeholder. This keeps the template self-contained and prevents the context from being lost-in-the-middle (the prompt engineering principle of primacy/recency).

The RUNTIME CONTEXT block contains:
```json
{
  "combined_complexity_score": <int 1-25 from Level 1>,
  "call_graph_risk_level": "<LOW|MEDIUM|HIGH>",
  "call_graph_hot_nodes_count": <int>,
  "call_graph_affected_methods_count": <int>,
  "call_graph_complexity_boost": <int>
}
```

`build_prompt_gen_input()` assembles this block from state fields that Pre-0 and Level 1 already populate.

## C. Verbose Output Design

### How the Existing Hooks Inject Output into the Conversation Stream

The hooks (post_tool_tracker, stop_notifier, pre_tool_enforcer) are Claude Code hooks that run as separate processes. They do NOT inject content into the Claude conversation stream. They operate outside of it:

- `post_tool_tracker` runs after every tool use and writes to session files + emits metrics
- `stop_notifier` runs after every Stop event (AI response completed) and does session maintenance
- `pre_tool_enforcer` runs before tool calls for policy enforcement

**None of these hooks stream into the conversation in real time.**

The mechanism that produces visible output in Claude Code is different:

- `print(..., file=sys.stderr)` in pipeline nodes writes to the terminal stderr
- Claude Code shows this terminal output in its console/terminal pane
- The pipeline uses this extensively: `print(f"[STEP {step_number:02d}] {step_label} - START", file=sys.stderr)`

### The Real Question: How Does a Subprocess Show Up in Conversation

For the orchestrator-agent subprocess call in v1.14.0, there are two options:

**Option 1: subprocess with stderr passthrough (recommended)**

Instead of `capture_output=True`, use `stdout=subprocess.PIPE, stderr=None` (or `stderr=subprocess.STDOUT` piped to a line-by-line reader). This lets the subprocess write directly to the parent process stderr, which Claude Code displays.

```python
# Pseudocode pattern
proc = subprocess.Popen(
    [sys.executable, str(orchestrator_script)],
    stdout=subprocess.PIPE,
    stderr=None,  # inherit parent stderr = visible in Claude Code terminal
    text=True,
    encoding="utf-8",
)
# Read stdout line by line to capture JSON result
result_lines = []
for line in proc.stdout:
    result_lines.append(line)
proc.wait()
```

The orchestrator script prints its progress to stderr (same `[STEP XX] - START/OK` pattern) and writes its JSON result to stdout.

**Option 2: Flag-file signaling (existing stop_notifier pattern)**

The stop_notifier uses flag files (`.session-start-voice-{PID}`) for signaling. This pattern is too indirect for real-time progress.

**Decision: Option 1 with stderr passthrough.**

The exact function to call is `subprocess.Popen` instead of `subprocess.run`. The `call_execution_script` helper uses `capture_output=True` which suppresses stderr. A new helper `call_streaming_script` (or a `streaming=True` flag on the existing helper) must be added.

### Which File Handles This and What Function to Call

1. In `level3_execution/helpers.py`: add `call_streaming_script(script_name, args, model_tier)` that uses `Popen` with `stderr=None` (inherited)
2. In `level3_execution/architecture/orchestrator-agent-caller.py`: the new script writes all progress to `sys.stderr` (visible) and writes JSON result to `sys.stdout`
3. In `nodes/step_wrappers_0to4.py`: the new Step 0 node uses `call_streaming_script` for the orchestrator call

**Print pattern for the orchestrator script:**
```python
print("[ORCHESTRATOR] Starting plan execution...", file=sys.stderr, flush=True)
print("[ORCHESTRATOR] Step 1/3: Analyzing requirements...", file=sys.stderr, flush=True)
# ... do work ...
print("[ORCHESTRATOR] Step 2/3: Building agent assignments...", file=sys.stderr, flush=True)
# ... do work ...
print("[ORCHESTRATOR] Step 3/3: Generating execution plan...", file=sys.stderr, flush=True)
# ... do work ...
print("[ORCHESTRATOR] DONE", file=sys.stderr, flush=True)
json.dump(result, sys.stdout)  # only JSON goes to stdout
```

The `flush=True` is critical on Windows to ensure lines appear immediately rather than being buffered.

## D. Risks and Decisions

### Risk 1 — State Field Pollution from Orchestrator Result

**Risk:** The orchestrator-agent subprocess returns a complex nested dict. If this is written to state wholesale as `orchestrator_result`, and Steps 8-14 try to read legacy fields (step5_skill, step7_execution_prompt, etc.) from it, there will be key conflicts or missing fields.

**Mitigation:** After `call_orchestrator_agent()` returns, a post-processing function `_map_orchestrator_result_to_state_fields()` maps the orchestrator result's fields into the existing step1/3/5/6/7 migration field names. This is exactly what the current Step 0 wrapper does with `setdefault` — v1.14.0 extends the same pattern. Steps 8-14 see the same state fields they already expect.

### Risk 2 — Subprocess Timeout for the Orchestrator Agent Call

**Risk:** The orchestrator-agent is itself calling an LLM (prompt-gen-expert result -> orchestrator). LLM calls can take 30-120 seconds. The existing `call_execution_script` has a `timeout=30` hardcoded. This will cause the orchestrator call to time out.

**Mitigation:** `call_streaming_script` (the new helper) must use a longer timeout or no timeout cap for the orchestrator call. Recommended: 300 seconds (5 minutes) for the orchestrator-agent call, 60 seconds for the prompt-gen-expert call. These become configurable via env vars `STEP0_PROMPT_GEN_TIMEOUT` and `STEP0_ORCHESTRATOR_TIMEOUT`.

### Risk 3 — Verbose Output Not Appearing in Claude Code

**Risk:** On Windows with subprocess stderr inheritance, there can be buffering issues where output appears in bursts rather than line by line.

**Mitigation:** Use `flush=True` on every `print()` in the orchestrator script. Set `PYTHONUNBUFFERED=1` in the subprocess env. Use `Popen` with line-by-line stdout reading so the parent is actively consuming output rather than waiting.

### What Stays Completely Unchanged

- Pre-0 node (`orchestration_pre_analysis_node`) — call graph scan and RAG lookup are identical
- Template fast-path — if `--orchestration-template` flag is given, Pre-0 still bypasses Step 0 entirely
- RAG hit path — if RAG hits at 0.85+, Pre-0 still bypasses Step 0 entirely
- Steps 8-14 — all GitHub, branch, impl, PR, closure, docs, summary steps are untouched
- All three hook scripts (pre_tool_enforcer, post_tool_tracker, stop_notifier) — untouched
- `_run_step()` wrapper in subgraph.py — the Step 0 node still goes through it for metrics/checkpointing

### Backward Compatibility Concerns

1. **RAG cache invalidation:** Step 0 now writes different state fields (orchestration_prompt, orchestrator_result) compared to v1.13.0. Existing RAG cached plans from v1.13.0 will not have these fields. On RAG hit, the cached plan bypasses Step 0 entirely anyway — no incompatibility. On RAG miss, Step 0 runs the new 2-call chain and RAG stores the result including the new fields. Backward compat is clean.

2. **Template fast-path:** Unchanged. The orchestration_template dict in state still bypasses Step 0 directly to Step 8.

3. **FlowState TypedDict:** New fields are `Optional[...]` so existing code that does not know about them is unaffected.

4. **Tests:** Tests that mock `step0_task_analysis` will break if they reference the old step function. Tests that test the Step 0 output fields will need updating only if they check for `orchestration_prompt` or `orchestrator_result` absence.

## Supplemental: Where build_prompt_gen_input Lives

`build_prompt_gen_input()` is a pure Python function (no LLM call) that should live in `nodes/step_wrappers_0to4.py` itself as a module-level helper, NOT in a separate architecture script. It:

1. Reads the template file from `level3_execution/templates/orchestration_system_prompt.txt`
2. Reads `state.user_message`
3. Builds the RUNTIME CONTEXT dict from `state.combined_complexity_score`, `state.call_graph_metrics`
4. Formats the template string with both injection points
5. Returns the assembled string

This keeps the assembly logic testable in unit tests without spawning subprocesses.

## ADR-001: Template Storage Format

**Decision:** `.txt` file at `level3_execution/templates/orchestration_system_prompt.txt`
**Rejected:** Python constant (poor editability), `.md` file (markdown ambiguity in prompts)
**Consequence:** Template changes do not require code commits, just file edits

## ADR-002: Subprocess Communication Pattern for Orchestrator Call

**Decision:** `subprocess.Popen` with `stderr=None` (inherited) and line-by-line stdout reading
**Rejected:** `subprocess.run(capture_output=True)` — suppresses stderr, no streaming; flag-files — too indirect
**Consequence:** Progress appears in Claude Code terminal pane in real time; new `call_streaming_script()` helper needed

## ADR-003: State Field Mapping

**Decision:** Post-call mapping function `_map_orchestrator_result_to_state_fields()` in step_wrappers_0to4.py
**Rejected:** Writing orchestrator result fields directly into new unique state field names (would require updating Steps 8-14)
**Consequence:** Steps 8-14 are completely insulated from v1.14.0 changes

## ADR-004: Timeout Configuration

**Decision:** New env vars `STEP0_PROMPT_GEN_TIMEOUT` (default 60s) and `STEP0_ORCHESTRATOR_TIMEOUT` (default 300s)
**Rejected:** Hardcoded timeouts (existing 30s would kill both calls)
**Consequence:** Operators can tune timeouts without code changes

## Interface Contract: Step 0 (v1.14.0)

```
FROM:      orchestration_pre_analysis_node (RAG MISS path)
TO:        step0_task_analysis_node (new v1.14.0 version)
INPUT:     state.user_message (str),
           state.combined_complexity_score (int 1-25),
           state.call_graph_metrics (dict with hot_nodes, leaf_nodes, complexity_boost)
OUTPUT:    state.orchestration_prompt (str) -- the filled orchestration prompt
           state.orchestrator_result (dict) -- execution plan from orchestrator agent
           state.step7_execution_prompt (str) -- populated from orchestrator_result for Steps 8-14
           state.step1_plan_required, step3_tasks_validated, step4_model,
           step5_skill/agent/skills/agents, step6_skill_ready/agent_ready (all from mapping)
ASSUMES:   Pre-0 has run; combined_complexity_score is in state from Level 1
MUST NOT:  Modify steps 8-14 state fields directly; call analyze_impact_before_change() again
           (Pre-0 already did it); read from RAG cache (that is Pre-0's responsibility)
```
